# Epic A — CRM data-model depth — design

**Date:** 2026-06-15
**Status:** approved-direction (user) — decisions controller-made
**Program:** GoHighLevel feature-parity, Epic A of N (foundational data-model layer)

## Goal

Add the foundational CRM data-model primitives GoHighLevel has and that the rest
of the parity program builds on: **custom fields**, **tags**, **saved segments**,
**duplicate detection + merge**, and **CSV import**. Each is workspace-scoped and
follows the existing `marketing` module conventions (controller/service/dto,
hand-authored Prisma migrations, outbox events, `@Audit`, soft references).

The `Lead` entity doubles as GHL's *contact* **and** *opportunity*. v1 targets
`Lead` only; the schema is enum-ready for future entities without a rewrite.

These five units are the bedrock for later epics: workflow triggers/actions,
campaign audiences, and the public API (Epic B) all consume them.

## Decisions (controller-made)

- **Custom-field storage = Hybrid.** Typed definitions in a `CustomFieldDef`
  table; values in a `Lead.customFields` JSONB column with a GIN index. Definitions
  give validation + UI metadata; JSONB keeps values queryable (Prisma JSON path
  filters) without an EAV join per field.
- `CustomFieldDef.key` (slug) and `.type` are **immutable after creation** — values
  are keyed by `key`, and re-typing existing values is unsafe. Renaming `label`
  is fine. Deleting a def = **archive** (values retained); hard-delete is a
  separate explicit action that strips the key from all rows.
- **Tags** are workspace-scoped, case-insensitive-unique, auto-created on
  assign-by-name (GHL behaviour). Tag↔Lead is a `LeadTag` join table.
- **Segments are DYNAMIC** (live-evaluated predicate). Static frozen lists are out
  of scope for v1. The segment predicate compiler is the **single source of truth**
  for audience selection — `Campaign.audienceFilter` is migrated onto the same DSL.
- **Merge is transactional, idempotent, audited, and emits an outbox event.**
  Children are re-parented to the canonical lead; the duplicate is tombstoned via
  `mergedIntoId` (not deleted) so history and links survive.
- **CSV import is async + batched** through the existing `ScheduledJob` queue
  (`kind = 'import.batch'`). Rows are persisted to `ImportJobRow`; the raw file is
  not stored beyond parsing.
- Migrations are **hand-authored SQL** (the dev/CI env runs DB-less; Prisma only
  needs to know the columns exist — same approach as `KnowledgeDoc.searchVector`).

### Non-goals (v1)

- Custom fields on entities other than `Lead` (enum-ready, not built).
- Static/frozen segment lists, segment-level analytics dashboards.
- Fuzzy/ML duplicate matching — v1 matches on normalized phone + email only.
- CSV *export*, scheduled/recurring imports, external ETL connectors.
- A drag-and-drop UI for the segment builder beyond a structured rule editor.

## Unit A1 — Custom fields

### Data model
```prisma
model CustomFieldDef {
  id          String   @id @default(uuid())
  workspaceId String
  entity      String   @default("LEAD") // LEAD (enum-ready: CONTACT|OPPORTUNITY)
  key         String   // slug, immutable, unique per (workspace, entity)
  label       String
  type        String   // TEXT|TEXTAREA|NUMBER|DATE|DATETIME|BOOL|SELECT|MULTISELECT|URL|PHONE|EMAIL
  options     Json?    // [{value,label}] for SELECT/MULTISELECT
  required    Boolean  @default(false)
  position    Int      @default(0)
  archived    Boolean  @default(false)
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  @@unique([workspaceId, entity, key])
  @@index([workspaceId, entity, archived])
  @@map("custom_field_defs")
}
```
```prisma
// added to model Lead:
customFields Json @default("{}") // { "<key>": value }
// migration adds: CREATE INDEX leads_custom_fields_gin ON leads USING GIN (custom_fields jsonb_path_ops);
```

### `CustomFieldsService`
- `list(workspaceId, {includeArchived})` / `create` / `update` (label, options,
  required, position only) / `archive` / `restore` / `hardDelete` (strips key from
  all leads in a transaction) / `reorder(ids[])`.
- `validateAndNormalize(workspaceId, input: Record<string,unknown>): Record<string,unknown>`
  — the integration seam. Loads non-archived defs, coerces each value to its type
  (number/date parsing, bool, URL/phone/email format), enforces SELECT options and
  `required`, drops unknown keys (or 400 — see DTO), returns the normalized map.
  Called by lead create + update before persist.
- `key` generated by slugifying `label` on create if not supplied; collision → 409.

### Controller — `marketing-custom-fields.controller.ts`
`@Controller('marketing/custom-fields')`, workspace JWT guard:
- `GET /` · `POST /` · `PATCH /:id` · `DELETE /:id` (archive) ·
  `POST /:id/restore` · `DELETE /:id/hard` · `POST /reorder`. All `@Audit`.

### Integration
- `marketing-leads.service` create/update: run `customFields` through
  `validateAndNormalize`; merge (not replace) on PATCH so partial updates work.
- Lead read responses include `customFields`.
- Emits `marketing.lead.customField.changed.v1` (per changed key) for workflows.

## Unit A2 — Tags

### Data model
```prisma
model Tag {
  id          String   @id @default(uuid())
  workspaceId String
  name        String
  nameLower   String   // maintained by service for case-insensitive uniqueness
  color       String?  // hex
  createdAt   DateTime @default(now())

  @@unique([workspaceId, nameLower])
  @@index([workspaceId])
  @@map("tags")
}

model LeadTag {
  leadId       String
  tagId        String
  assignedAt   DateTime @default(now())
  assignedById String?

  lead Lead @relation(fields: [leadId], references: [id], onDelete: Cascade)

  @@id([leadId, tagId])
  @@index([tagId])
  @@map("lead_tags")
}
```
`Lead` gains `tags LeadTag[]`. `Tag` gains `@@index` for member counts.

### `TagsService`
- `list(workspaceId)` (with member counts) / `create` / `rename` / `setColor` /
  `delete` (cascades `LeadTag`).
- `assign(leadId, tagNamesOrIds[])` — auto-creates tags by name (case-insensitive
  lookup on `nameLower`), idempotent. `unassign(leadId, tagIds[])`.
- `bulkAssign(workspaceId, leadIds[], tagIds[])` / `bulkUnassign`.
- Emits `marketing.lead.tag.added.v1` / `marketing.lead.tag.removed.v1`.

### Controller — `marketing-tags.controller.ts`
`@Controller('marketing/tags')`: `GET /` · `POST /` · `PATCH /:id` · `DELETE /:id`.
Plus on the leads controller: `POST /leads/:id/tags`, `DELETE /leads/:id/tags/:tagId`,
`POST /leads/tags/bulk` (bulk assign/unassign). All `@Audit`.

## Unit A3 — Saved segments (keystone)

### Data model
```prisma
model Segment {
  id              String   @id @default(uuid())
  workspaceId     String
  name            String
  description     String?
  kind            String   @default("DYNAMIC") // DYNAMIC (v1)
  definition      Json     // predicate tree (grammar below)
  lastCount       Int?
  lastEvaluatedAt DateTime?
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt

  @@index([workspaceId])
  @@map("segments")
}
```

### Predicate DSL (the contract)
A JSON tree. A **group** has `op` (`and`|`or`) + `children[]`. A **leaf** has
`field`, `cmp`, and (usually) `value`:
```jsonc
{ "op": "and", "children": [
  { "field": "status",        "cmp": "in",   "value": ["NEW","CONTACTED"] },
  { "field": "city",          "cmp": "eq",   "value": "Istanbul" },
  { "field": "createdAt",     "cmp": "gte",  "value": "2026-01-01" },
  { "field": "cf:budget",     "cmp": "gte",  "value": 1000 },     // custom field
  { "field": "tag",           "cmp": "has",  "value": "<tagId>" },// has this tag
  { "op": "or", "children": [
    { "field": "source",      "cmp": "eq",   "value": "WEBSITE" },
    { "field": "aiScore",     "cmp": "gte",  "value": 70 }
  ]}
]}
```
- **Fields:** native whitelisted `Lead` columns (`status, city, region, source,
  businessType, priority, aiScore, createdAt, nextFollowUp, convertedAt,
  emailOptOut, smsOptOut, waOptOut`), `cf:<key>` (custom field, type-checked against
  its def), `tag` (membership).
- **Comparators:** `eq, ne, in, nin, gt, gte, lt, lte, contains, startsWith,
  isSet, isNotSet, between` (+ `has`/`hasNot` for `tag`). Per-type comparator
  whitelist enforced at validation.

### `SegmentCompilerService`
- `compile(workspaceId, definition): Prisma.LeadWhereInput` — recursively maps the
  tree: native fields → column predicates; `cf:<key>` → Prisma JSON path filter
  (`{ customFields: { path:[key], gte: v } }`); `tag` → `{ tags: { some/none:{ tagId } } }`.
  Always AND-pins `workspaceId` and `mergedIntoId: null` (exclude tombstones).
- `validate(workspaceId, definition)` — structural + field-whitelist + per-type
  comparator + custom-field-exists checks; throws 400 with a path to the bad node.
- Depth/size caps (≤ 6 levels, ≤ 100 nodes) to bound query cost.

### `SegmentsService`
- CRUD (validate `definition` on write). `preview(definition)` → `{ count, sample }`
  without saving. `count(id)` / `members(id, page)` → run the compiled `where`.
  `count` stamps `lastCount`/`lastEvaluatedAt`.

### Controller — `marketing-segments.controller.ts`
`GET /marketing/segments` · `POST /` · `PATCH /:id` · `DELETE /:id` ·
`POST /preview` · `GET /:id/members`. `@Audit` on writes.

### Integration — unify campaign audience
- `Campaign.audienceFilter` is migrated to the **same DSL**; campaign send resolves
  recipients via `SegmentCompilerService.compile`. A campaign may reference a
  `segmentId` *or* carry an inline `definition`. (Back-compat: a one-time data
  shim converts any legacy `audienceFilter` shape; if none exists in prod yet,
  this is a no-op documented in the migration.)
- Workflows can use `lead.entersSegment` as a trigger (see cross-cutting).

## Unit A4 — Duplicate detection + merge

### Data model
```prisma
// added to model Lead:
phoneE164       String?   // normalized phone for dedup/match
emailNormalized String?   // lower(trim(email))
mergedIntoId    String?   // tombstone → canonical lead id (soft, self-ref)
mergedAt        DateTime?
// migration adds indexes: (workspaceId, phoneE164), (workspaceId, emailNormalized), (mergedIntoId)
```
`phoneE164`/`emailNormalized` are maintained on every lead write (create/update)
and backfilled by the migration.

### `LeadDedupeService`
- `findDuplicates(workspaceId, {strategy})` — groups active leads (`mergedIntoId
  null`) sharing `phoneE164` and/or `emailNormalized` into candidate clusters;
  returns clusters with a suggested canonical (oldest / most-complete).
- `merge(workspaceId, canonicalId, duplicateIds[], resolution?)` — in a single
  `$transaction`:
  1. **Guard:** all leads same workspace, all active, canonical ∉ duplicates,
     not two `convertedTenantId`-set leads (409 conflict — can't merge two
     customers).
  2. **Re-parent children** to canonical: `LeadActivity, MarketingTask, LeadOffer,
     Commission, SalesCall, Conversation, ContactIdentity, Booking, Review,
     Invoice, WorkflowRun, VoiceCall`. `CampaignRecipient` has a
     `@@unique([campaignId, leadId])` — on collision, drop the duplicate's row
     (keep canonical's).
  3. **Union** `customFields` (canonical wins per key unless `resolution` says
     otherwise) and `LeadTag` (dedupe on conflict).
  4. **Fill blanks** on canonical from duplicates for null scalar fields (or apply
     explicit `resolution` map).
  5. Tombstone each duplicate: `mergedIntoId = canonicalId`, `mergedAt = now`.
  6. `@Audit` + emit `marketing.lead.merged.v1` (canonicalId, mergedIds).
- Idempotent: re-merging an already-tombstoned id is a no-op.

### Controller — on the leads controller
`GET /marketing/leads/duplicates` (clusters) · `POST /marketing/leads/merge`
(`{canonicalId, duplicateIds, resolution?}`). `@Audit`.
- All lead **read/list** queries add `mergedIntoId: null` so tombstones disappear
  from the pipeline; direct `GET /leads/:id` of a tombstone returns it with a
  `mergedInto` pointer (so old links resolve).

## Unit A5 — CSV import

### Data model
```prisma
model ImportJob {
  id          String   @id @default(uuid())
  workspaceId String
  filename    String
  status      String   @default("MAPPING") // MAPPING|RUNNING|DONE|FAILED|CANCELLED
  mapping     Json?    // { csvHeader -> {field} } incl. cf:<key>, tag, __skip
  dedupePolicy String  @default("SKIP")    // SKIP|UPDATE|CREATE
  total       Int      @default(0)
  processed   Int      @default(0)
  created     Int      @default(0)
  updated     Int      @default(0)
  skipped     Int      @default(0)
  failed      Int      @default(0)
  errors      Json?    // [{row, message}] (capped sample)
  createdById String?
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt

  @@index([workspaceId, status])
  @@map("import_jobs")
}

model ImportJobRow {
  id          String   @id @default(uuid())
  importJobId String
  rowIndex    Int
  raw         Json
  status      String   @default("PENDING") // PENDING|DONE|SKIPPED|FAILED
  leadId      String?
  error       String?

  @@index([importJobId, status])
  @@map("import_job_rows")
}
```

### `ImportService`
- `upload(workspaceId, file)` — parse CSV (add `csv-parse` dep; stream, cap 50k
  rows / 10 MB), create `ImportJob` (status MAPPING) + `ImportJobRow` per row,
  return headers + a **suggested mapping** (fuzzy header→field match against native
  fields + custom-field labels + a `tags` column).
- `commit(jobId, {mapping, dedupePolicy})` — validate mapping, set status RUNNING,
  enqueue `ScheduledJob{ kind:'import.batch', payload:{ jobId, offset:0 } }`.
- `processBatch(jobId, offset)` (worker, batches of ~500): for each row map →
  `CustomFieldsService.validateAndNormalize` → resolve tags → dedupe via
  `LeadDedupeService` keys → apply `dedupePolicy` (SKIP existing / UPDATE existing /
  CREATE always) → upsert lead → stamp row + counters. Re-enqueue next offset until
  done; then status DONE + final `stats`.
- Errors are per-row (never abort the whole job); capped sample stored in `errors`.

### Controller — `marketing-imports.controller.ts`
`POST /marketing/imports` (multipart upload → headers+suggested mapping) ·
`GET /marketing/imports/:id` (status/report) · `POST /marketing/imports/:id/commit`
· `GET /marketing/imports` (history). `@Audit` on upload/commit.

## Cross-cutting

### Workflow engine integration (wires Epic A into automation)
Additively register on the existing workflow trigger/action registry:
- **Triggers:** `tag.added`, `tag.removed`, `customField.changed`,
  `lead.entersSegment` (evaluated when a lead changes + matches a segment it
  didn't before). Backed by the domain events emitted above.
- **Actions:** `addTag`, `removeTag`, `setCustomField`. They call `TagsService` /
  `CustomFieldsService` inside the workflow run context.
- Exact registration follows the engine's existing pattern (read
  `marketing/workflows/` during implementation; this is additive — no DSL break).

### Frontend (`frontend/src/pages/marketing/`)
- **Settings → Custom Fields**: list/create/edit/reorder/archive field defs.
- **Settings → Tags**: manage tags (name, color, member counts, merge/delete).
- **Segments**: list + a structured rule builder (group/leaf editor over the DSL)
  with live `preview` count.
- **Leads detail / create-edit**: a Custom Fields section (rendered from defs) +
  a tag chip editor.
- **Leads list**: tag filter + "select a segment" + bulk tag assign +
  **Duplicates** review/merge screen (cluster view → pick canonical → merge).
- **Import wizard**: upload → column-mapping step → dedupe-policy + preview → run →
  live progress + report.
- **Campaigns**: audience step switches to the segment selector / inline rule
  builder (same component as Segments).
- Reuse `features/marketing/` API clients + React Query patterns; i18n keys (tr/en).

### Env
- No new required env. Add config for upload limits
  (`IMPORT_MAX_ROWS=50000`, `IMPORT_MAX_BYTES=10485760`) with safe defaults.
- New dep: `csv-parse` (backend).

## Testing

**Unit (Jest, backend):**
- `custom-fields.service.spec` — type coercion per type, SELECT-option + required
  enforcement, unknown-key handling, key/type immutability, slug collision (409),
  hard-delete strips keys.
- `tags.service.spec` — case-insensitive uniqueness, auto-create on assign,
  idempotent assign/unassign, bulk ops, delete cascades.
- `segment-compiler.service.spec` — every comparator → correct Prisma `where`;
  `cf:` JSON path; `tag` some/none; nested and/or; workspace + tombstone pinning;
  validation rejects bad field/comparator/depth.
- `lead-dedupe.service.spec` — cluster grouping; merge re-parents every child type;
  CampaignRecipient unique-collision handled; customFields/tag union; tombstone +
  idempotent re-merge; two-converted-leads guard (409).
- `import.service.spec` — parse + suggested mapping; per-row validation;
  SKIP/UPDATE/CREATE policies; batch re-enqueue; per-row error isolation; caps.

**E2E (Jest, `backend/test/e2e/`):**
- `custom-fields.e2e-spec` — CRUD + lead create with custom values + validation 400.
- `segments.e2e-spec` — create segment, preview count, members reflect tag/cf
  predicates.
- `lead-merge.e2e-spec` — seed dup cluster, merge, assert children re-parented +
  tombstone hidden from list.
- `import.e2e-spec` — upload → commit → drain `import.batch` → report counts.

**Architecture-fitness:** new services/queries enforce `workspaceId` scoping
(extend the existing fitness test). **The existing 85 unit + 13 e2e specs must
stay green** — no behavioural change to current endpoints beyond additive
`customFields`/`tags` in lead payloads and the `audienceFilter` DSL unification
(covered by an updated campaign-audience test).
